## Discussion

**The premise, examined (opening reframe).** The instinct — narrow the surface instead of guarding a wide one — is right and matches our own precedents (loderite composition-root 2026-07-20; kienzler www-copy 2026-06-23). But two caveats: (1) a narrow surface only *contains* if it is the **only grant**, not a permission-filter on Bash — `deny-probe-5937` found `permissionMode=auto` denies nothing and `--disallowedTools` is evadable via ToolSearch→Monitor; (2) the relay tools must be arg-safe or the whitelist re-opens injection through the tool. So the real work is placing the containment boundary correctly and *proving* it, not assuming it.

**Substrate mechanics (Zommuter's question).** "Can `drain-driver.mjs` orchestrate agents — and if so why keep a Workflow at all?" Yes: a host node process dispatches agents via `claude -p` subprocesses (`DRAIN_ROUND_CMD` is exactly that seam). The trade is a mirror, not a hierarchy: Workflow gives managed concurrency / lifecycle / resumability / budget / progress **free** but traps you in a sandbox where a mechanical step must be wrapped in an LLM agent; the host driver runs bash directly and keeps the queue outside any LLM (436c) but **re-implements** Workflow's runtime by hand (drain-driver already hand-rolls heartbeat id:f9d2, quota gating id:838d, event emission id:dd1e, the stop predicate). 🎛️ Orla: the sandbox limit *only ever bit the mechanical steps* — the genuine LLM leaf units (executor, reviewer) are agents anyway and `agent()` handles them in-billing.

**The ToS axis (Zommuter).** Host-driver agent dispatch is `claude -p` — the exact surface the postponed-but-not-abandoned no-subscription / API-credits-only change would hit (`sdk-billing-deferred`: "KEEP hedges"; `probe-tos-path` id:2d01). Workflow `agent()` stays inside sanctioned interactive-session billing. Mitigant: `DRAIN_ROUND_CMD` is an env seam, so dispatch is already swappable. This softens the host driver's "it can dispatch agents too" advantage.

**Independent verification (this session).** `mechanical-proxy.py` exists (23.5 KB, 2026-07-19). `model:"bash"` runs the shell command locally with **zero real model calls** (ToS posture: declining-to-send ≠ vendor-substitution). Its `_command_allowed` is genuinely strict: refuses command/process substitution (`$(…)`, backticks, `<(…)`), refuses unquoted redirection, splits every compound command on `| || && ; & \n` and requires **each** segment's leader to be safe-plumbing or a **realpath-identity-pinned** relay script (basename look-alikes rejected), ≥1 real relay script present, fail-open to the real model on anything unclear. 8ba1/9f1b are design-decided but unbuilt; no executor is working them in this repo (only live claim is a loderite executor + this meeting).

**"Even stricter, if sensible" (Zommuter) — a real residual found.** 🛡️ Fenn: the allowlist permits a plumbing leader (`cat`, `echo`, `printf`) in **any** `;`-separated segment, not only as a non-leading pipe stage as line 184 documents. So `cat ~/.claude/.credentials.json ; <pinned>/claim.sh peek` passes (cat = safe-plumbing, claim.sh = pinned relay, ≥1 relay present) and the proxy returns the command's stdout **verbatim — including the 0600 OAuth credential file.** Latent today (relay-loop.js composes these strings deterministically) but live under the proxy's *own* threat model (a model composes the `model:"bash"` command). Sensible fix: a mechanical hop is *one simple command or one pipeline that **ends** in a relay script* — refuse `;`/`&&`/`&`/newline, require the pipeline output stage to be a relay script; verify against relay-loop.js's real hops before tightening.

**Probe scope (Zommuter's steer).** The probe must confirm a **Workflow actually works** end-to-end, not curl the proxy in isolation: launch a real Workflow with `ANTHROPIC_BASE_URL`→proxy that does (a) an `agent(…, {model:"bash"})` hop → intercepted, zero upstream, stdout returned; and (b) one *normal* `agent()` → **its traffic must appear at the proxy** and relay upstream. (b) is the decisive premise — if a plain `agent()` does not transit the proxy, `agent()` does not use the global base URL, consolidation is off, and the off-Workflow line stands.
