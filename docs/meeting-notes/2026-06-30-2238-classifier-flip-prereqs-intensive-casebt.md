- User's rule — *"handoff iff promotable TODOs; else human (lane-triage) or idle"* — changes the cost tier of the same behaviour, not the behaviour.
- **Load-bearing constraint (Sage):** the anti-gaming invariant (surface backlog NEVER silently hidden as idle) must survive. `human` must be loud AND must still file to the decision-queue, and the relocated filing must be a *forced + logged + tested* mechanical step — a deterministic detector whose resolution silently no-ops is the exact relay-2026-06-30 anti-pattern (unpromoted-scan found 17, promoted 0).

The user's intuition lands as "operative only on dispatchable lanes," not literally "amplification of HARD."

- The **lint** is the actual false-positive (not the 9321 tag): it rejects legit advisory hands-intensive AND legit ROUTINE-intensive. id:a707's gather exclusion already neutralises the dispatch hazard, so the LOUD-reject is redundant *and* wrong.

- **id:5eb3 — case-b splits into three deterministic branches.** `classify-verdict.sh`: `promote>0 → handoff` (real ROADMAP-promotion work for Opus); `promote==0 ∧ surface>0 → human` (no apex dispatch); `promote==0 ∧ surface==0 → idle` (unchanged). The surface→decision-queue filing **relocates out of the Opus handoff handler into a forced, logged, tested mechanical step** wired where the loop consumes the `human` verdict, preserving loud surfacing + the id:47f1 case-g exclusion. Flip the case-b test (`tests/test_classify_verdict.sh:41`) and update `handoff.md`'s case-g section. **Out of scope:** the `promote>0` handoff path (unchanged); any change to `unpromoted-scan`'s promote/surface split.
- **id:5ac6 — flag + invariant + fail-closed assertion.** `classify-verdict.sh` copies gather's `top_intensive` to an `intensive:"<resource>"` field beside an unchanged `verdict`. Lock the invariant `intensive!="" ⟹ verdict∈{execute,hard}` in `test_classify_verdict.sh`. Add a **fail-closed pre-dispatch assertion** in `relay-loop.js`: never spawn an executor/apex child on a unit with `intensive` set unless `ALLOW_INTENSIVE` — skip + surface loudly, never OOM-dispatch. Keep the existing loop partition + `resource:` claim. Three layers; the classifier+test is the new durable guard. **Out of scope:** changing the `resource:` claim or serially-alone scheduling; auto-running intensive work.
- **id:9062 (NEW) — roadmap-lint case (d) realigned to the operative-only-on-dispatchable-lanes model.** ACCEPT `[INTENSIVE]` on `[ROUTINE]` or `[HARD — pool]` (operative); ACCEPT on human lanes (`hands`/`meeting`/`decision gate`/`@manual`) as advisory — **no violation**; REJECT only `[INTENSIVE]` with no recognised lane. Fix the contradictory `roadmap-lint.sh:80` comment and the db39 doc so lint / gather / hard-lanes.md finally agree; update `test_roadmap_lint`. **it-infra id:9321 keeps both tags — no it-infra edit** (the fix is entirely here; a `/relay human` session is live on it-infra and must not be touched). **Out of scope:** any soft-advisory note line (the lighter accept-silently option was chosen).
- **Doctrine restatement (lands in `hard-lanes.md` / conventions):** `[INTENSIVE — <resource>]` is operative (serial scheduling, exclusive `resource:` claim, `--intensive` gate, classifier flag) **only on relay-dispatchable lanes** (`[ROUTINE]`, `[HARD — pool]`); on human lanes it is an advisory OOM note and is mechanically inert. Not "amplification of HARD" (routine-intensive is operative).
