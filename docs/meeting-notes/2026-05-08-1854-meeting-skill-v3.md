## Discussion

### Item 1 — Effort-estimate methodology

**Lexi (onboarding — DMAIC lens):** Six Sigma's first move on any pessimistic estimate is Measurement System Analysis. The two candidate independent variables: *calendar time* (current default — wrong, since Claude work is bursty) and *session-units* (closer, but heterogeneous). The right unit is the one whose distribution is narrowest per task class.

Apply DMAIC compactly:
- **D**efine: today's estimates conflate *observation windows* with *implementation effort*. Only the latter is pessimistic.
- **M**easure: data already exists — `~/.claude/projects/*.jsonl` per session, `**Started:**` in meeting notes for brackets. Today gave us 7 free data points.
- **A**nalyse: today's mean session is ~150 jsonl turns ≈ ~470KB ≈ ~120k tokens. Max-5x → ample headroom for 4–6 such sessions per 5h block.
- **I**mprove: switch from calendar-days to **Claude-session-equivalents** + optional token annotation. Keep calendar dates only for observation windows / external deadlines.
- **C**ontrol: re-measure after 5 meetings; if CV per task class still > 50%, refine task taxonomy.

**Archie:** The offending phrases are triggers ("due 2026-05-22") and observation windows ("4-week observation"), not implementation estimates per se. The actual bug is in *action items* where "a few sessions of work" gets framed as "could take days." Fix the framing, not the trigger semantics.

**Riku:** Failure mode: on Max-5x the resource is quota windows. A pessimistic estimate mis-schedules what fits in one quota block. The fix should inform *batching*, not provide reassurance.

**Petra:** N=2 check: (a) user scoping take-on; (b) model sequencing items in no-arg mode. Two real consumers. Proceed.

**Sage:** Avoid new instrumentation. Stop-hook `transcript_path` discovery means we can post-hoc compute any session's cost. Pair with `**Session:**` field from v2 → one-shot `cost-of.sh <session-id>` script. Skill stays prompt-only.

**Lexi:** After ~10 meetings we publish a rule-of-thumb table in `format.md` (v0 seeded today). Converts effort from guess to regression problem.

**Archie:** Two outputs: (a) rule in the skill — estimates in `s` (session-equivalents) + `~Nk tok`, never `days` for implementation; (b) the lookup table, updated when model notices significant deviation.

**Riku:** Revisit trigger: 5 more meetings. If unit stays stable but values shift >2× → table wrong, no methodology change. If unit itself looks wrong → revisit. Trigger by event count, not date.

**Petra (closing):** In scope: language change, cost script, table v0, Stop-hook telemetry (by user override of N=2). Out of scope: dashboards, regression analysis.

### Item 3 — User profiling for personas

**Sage:** Two requests: (a) profile capture — durable storage; (b) profile use — sparse pre-emption + future autopilot. (a) is the foundation.

**Riku:** Failure modes: false profile (one-off → applied broadly), stale profile, annoying pre-emption, autopilot drift. False/stale = trust failures; annoying = ergonomic; autopilot drift = agency loss.

**Lexi:** Today's candidate signals: override patterns (broader scope than Petra's N=2, willingness to override consensus); decision style (picks combinations, prefers concrete data, asks for event-based re-evaluation triggers); domain affinities (Six Sigma fluent). Need a discipline for what counts.

**Petra:** Storage: (i) `feedback`-type memory — wrong category (rules, not traits); (ii) `user`-type auto-memory — natural fit; (iii) new `~/.claude/skills/meeting/user-profile.md` — meeting-scoped, nothing leaks to other contexts. Use both (ii) and (iii) in a two-tier approach.

**Archie:** Tier 1 (`user-profile.md`) for meeting-specific observations; Tier 2 (auto-memory `user`-type) for durable cross-context facts. Both go through end-of-meeting classification `[save to user-profile / save to user-memory / discard]`.

**Riku:** Profile entries use **Why** (evidence) + **How to apply** fields — same discipline as feedback memories. Without the why, can't judge if a pattern still holds.

**Sage:** Pre-emption: (α) always-on = bloat; (β) sparse, justified = only when contradicted by a high-confidence eligible fact; (γ) opt-in per project. (β) is the right default. User said "occasionally if sensible."

**Petra:** (β) + asymmetry — Riku handles most pre-emption; Archie/Petra/Sage rarely do.

**Lexi:** Schema is autopilot-ready via `Pre-emption-eligible: yes/no` and `Confidence` fields. Autopilot itself is deferred.

**Riku:** Revisit after 5 meetings with profile entries: miss (pre-empted, user disagreed) → tighten confidence threshold; never used → criteria too strict.

**Archie (concrete proposal):** `user-profile.md` loaded at setup step 6. Per-entry format: `## <trait>`, `**Observation:**`, `**Why (evidence):**`, `**Confidence:**`, `**Pre-emption-eligible:**`. End-of-meeting adds profile classification step. Pre-emption fires only when all three: eligible, confidence ≥ med, contradicted by proposal. Riku ≫ others.
